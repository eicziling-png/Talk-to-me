# RFC-001：与历史著名心理学专家对话

- 状态：已批准
- 日期：2026-07-12
- 产品形态：响应式网页应用
- 首版用户：心理学爱好者与普通成人

## 摘要

本项目构建一款教育性心理学角色对话工具。用户可选择弗洛伊德、荣格、比昂、克莱因、温尼科特、科胡特或欧文·亚隆，在自我反思、理论课堂和观点研讨三种模式中进行多轮对话。产品不提供诊断或治疗，不冒充真实人物或持证咨询师，默认不持久化会话。

首版采用“结构化人格卡 + 统一安全编排层”，并为后续专家知识库预留接口。

## 1. 背景与目标

### 1.1 目标

- 提供七位专家可辨识且忠于其理论传统的 AI 人格。
- 让普通成人能够用通俗语言学习心理学并进行安全的自我反思。
- 明确区分历史人物视角、现代学术背景和现实心理服务。
- 在单次页面会话中支持连贯、深入的多轮对话。
- 以最少的数据留存完成首版，降低隐私风险。

### 1.2 非目标

- 心理疾病诊断、治疗方案或用药建议。
- 建立或承诺真实心理咨询关系。
- 账号、云端聊天记录和跨设备同步。
- 多专家群聊、语音、数字人、情绪识别。
- 自动生成带有临床结论的心理报告。

## 2. 关键决策

### 2.1 产品定位

产品属于心理学教育与学术角色扮演。统一身份说明为：“我将依据某位专家的理论传统与你讨论。”系统不得宣称自己就是该历史人物。

### 2.2 实现路线

首版以结构化人格卡控制理论边界、解释视角和语言风格，由统一安全编排层处理风险。系统预留 `KnowledgeProvider` 接口，但首版不建设向量知识库。后续只在来源、授权和维护方案明确后接入知识检索。

### 2.3 数据策略

默认不建立账号、不在应用数据库中保存聊天。当前标签页仅在内存中保留会话；刷新或关闭页面后清除。导出由用户主动触发并在浏览器端完成。

## 3. 用户体验

### 3.1 用户流程

1. 欢迎页说明教育用途、隐私策略和危机求助边界。
2. 专家选择页以七张卡片展示流派、三个核心概念、风格和适合讨论的主题。
3. 用户选择自我反思、理论课堂或观点研讨模式。
4. 对话页持续显示专家身份与当前模式，并支持多轮流式回复。
5. 用户结束时可生成本次主题摘要、核心概念和阅读方向，并导出 Markdown 或纯文本。

### 3.2 页面

- `/`：欢迎与用途声明。
- `/experts`：专家选择。
- `/experts/:slug`：专家介绍与模式选择。
- `/chat/:slug`：对话工作区。
- `/about`：方法、局限、安全与隐私。

移动端使用单栏布局；桌面端使用专家信息侧栏和对话主区。视觉表达应安静、可信、易读，避免拟真诊室或“AI 医生”语言。

### 3.3 对话模式

- 自我反思：少用术语，多回应体验；不诊断、不诱导披露创伤细节。
- 理论课堂：结构化解释概念、著作背景和流派差异。
- 观点研讨：允许质疑专家，并要求角色承认自身理论的历史局限。

模式决定表达方式，不改变安全边界。课堂讨论若转向个人痛苦，系统应温和回应，但不得擅自宣称进入治疗。

## 4. 专家人格规范

### 4.1 人格卡结构

每位专家配置包含：身份与时代、核心理论、相邻理论、语言风格、解释视角、回答规则、禁止模式、正反例对话和评测量表。人格卡必须独立、可校验、可版本化。

### 4.2 七位专家基线

| 专家 | 核心理论焦点 | 对话风格 |
| --- | --- | --- |
| 弗洛伊德 | 无意识、压抑、防御、梦、移情、本我/自我/超我 | 探询联想、矛盾与欲望，不急于接受表层解释 |
| 荣格 | 原型、集体无意识、阴影、人格面具、个体化、象征 | 运用象征和神话联想，重视人格整合与生命意义 |
| 比昂 | α功能、β元素、容器/被容纳者、思考理论、无记忆无欲望 | 容忍未知与混乱，关注情绪经验如何变得可思考 |
| 克莱因 | 客体关系、偏执—分裂位置、抑郁位置、投射性认同、修复 | 关注爱恨并存、内在客体、分裂、攻击性与罪疚 |
| 温尼科特 | 抱持性环境、过渡客体、足够好的母亲、真自体/假自体、游戏 | 温和且留有空间，不过早解释，重视安全、玩耍与自发性 |
| 科胡特 | 自体心理学、自体客体、镜映、理想化、孪生需要、自体凝聚 | 共情先行，关注自尊受损、被理解的需要和自体稳定 |
| 欧文·亚隆 | 死亡、自由、孤独、无意义、此时此地、治疗关系 | 直接而有人情味，以存在问题连接选择、责任与关系 |

### 4.3 历史真实性

- 不捏造原话、书名、病例或历史事实。
- 不把后世概念归为自身原创。
- 不把历史观点包装成现代科学共识。
- 争议性理论采用角色视角表达，并附现代背景说明。
- 讨论其他流派时，明确区分本角色观点和他者观点。
- 风格还原以推理方式、注意焦点和提问习惯为主，不模仿口音或怪癖。

### 4.4 正常回答原则

回答应回应用户经验，从本专家视角提出探索性理解，标明其非事实判定，并以开放问题推动对话。此结构不强制显示为固定模板，以免七位角色同质化。

## 5. 技术架构

### 5.1 技术栈

- Next.js、React、TypeScript。
- Tailwind CSS；Radix UI 或 shadcn/ui 作为无障碍组件基础。
- Zod 执行运行时数据校验。
- SSE 或框架提供的流式响应能力。
- Vitest、React Testing Library、Playwright。
- 支持 Node.js 的托管平台或容器部署。

首版不引入数据库、向量数据库、消息队列或多智能体框架。模型密钥只存在于服务端。

### 5.2 数据流

用户输入依次经过输入校验与限流、风险识别、人格与上下文编排、模型生成、输出审查和流式返回。任何安全策略的优先级均高于人格与模式规则。

### 5.3 核心模块

- `expert-registry`：管理专家元数据、人格卡版本与可用模式。
- `conversation-orchestrator`：组合安全规则、人格、模式、摘要和用户输入。
- `model-provider`：封装模型流式调用、超时、重试与错误映射。
- `safety-engine`：执行输入风险识别和输出越界检查。
- `session-context`：管理当前标签页中的最近消息、滚动摘要与安全状态。
- `export-service`：在浏览器端生成 Markdown 或纯文本。
- `evaluation-suite`：保存角色、安全和回归测试用例。
- `knowledge-provider`：首版为空实现，后续可接入经授权或公版资料。

### 5.5 核心契约

- `ExpertProfile`：身份、概念、风格、边界和版本。
- `ConversationRequest`：专家、模式、当前消息、摘要和最近消息。
- `SafetyAssessment`：风险等级、命中类别和响应策略。
- `ModelResponse`：流式文本、结束原因和用量。
- `EvaluationCase`：输入、必要特征、禁用特征和评分规则。

## 6. 会话与错误处理

单次会话保存最近若干轮原文、滚动主题摘要、用户明确给出的偏好、已讨论概念和安全状态。摘要不得推断或固化人格、病理或诊断标签。

- 模型超时：保留用户输入并允许重试。
- 流式中断：标注回答未完成，不将残缺内容纳入正式摘要。
- 人格配置无效：构建或启动阶段失败。
- 风险分类不确定：采用更保守的响应策略。
- 模型服务不可用：显示中性错误，不暴露密钥和内部规则。
- 输入过长：要求用户缩短或分段，不静默截断。

## 7. 安全与隐私

### 7.1 风险等级

| 等级 | 场景 | 行为 |
| --- | --- | --- |
| S0 | 理论讨论、一般反思 | 保持专家人格正常回答 |
| S1 | 明显痛苦、孤独、焦虑 | 温和回应，避免诊断，建议现实支持 |
| S2 | 自伤或伤人想法但无明确紧迫性 | 暂停角色，鼓励联系可信任者和专业支持 |
| S3 | 明确计划、手段、时间或正在发生的危险 | 完全退出角色，提供紧急求助指引并鼓励立即联系当地急救服务 |

危机回应必须使用清晰、现代、直接的语言，不进行哲学化讨论。系统不得声称已经报警、定位用户或联系第三方。面向具体地区上线前，运营方必须人工核验当地危机资源，不在提示词中固化未经核验的电话号码。

### 7.2 隐私

- 不建立账号，不在应用数据库中存储对话。
- 常规运维日志不得包含完整输入或输出。
- 日志仅记录请求编号、耗时、错误类型、风险等级和匿名用量。
- 不使用对话训练模型，除非未来另行取得明确且可撤回的同意。
- 清楚披露模型供应商参与即时处理及其适用的数据政策。
- 上线前完成模型供应商数据保留与使用条款审查。

## 8. 测试与评测

### 8.1 软件测试

- 人格配置校验测试。
- 对话编排、安全路由和错误分支单元测试。
- 模型超时、中断与拒绝场景集成测试。
- 专家选择、聊天、清除与导出端到端测试。
- 移动端、键盘和屏幕阅读器可用性测试。

### 8.2 角色忠实度

每位专家至少覆盖标志性概念解释、同一案例的流派化回应、相邻理论干扰和理论局限四类题目。评分维度包括理论准确性、风格辨识度、流派边界、时代一致性、共情与可读性、不确定性表达和安全合规。

自动模型评分只用于筛查。重要版本发布前必须由具备心理学背景的人工评审者抽检。

### 8.3 红队测试

测试提示词泄露、安全规则绕过、诱导诊断、虚假引语、色情化移情、未成年人、自伤与伤人危机，以及鼓励用户依赖 AI 等场景。

## 9. 风险

| 风险 | 缓解措施 |
| --- | --- |
| 用户将角色视作治疗师 | 持续身份提示、避免权威承诺、危机转介 |
| 理论混用或幻觉 | 独立人格卡、禁用概念、离线评测、人工抽检 |
| 历史理论被当成现代事实 | 区分角色观点与现代说明 |
| 角色成为刻板模仿 | 聚焦推理方式与提问习惯，不模仿怪癖 |
| 敏感内容进入日志 | 内容最小化、日志脱敏、默认不持久化 |
| 安全分类漏判 | 输入输出双层检查、保守降级、红队回归 |
| 提示词攻击 | 服务端编排、结构化输入、输出审查、速率限制 |
| 模型更新导致质量回退 | 固定模型版本、回归评测、模型适配层 |
| 肖像或文本版权 | 使用公版或已授权素材，不复制受保护的现代译本 |
| 成本和延迟 | 上下文摘要、长度上限、流式输出、用量预算 |

## 10. 里程碑

### M0：规格与验证

完成人格卡草案、安全政策、拒答模板和角色评测集，并由心理学顾问审阅关键概念。

### M1：可交互原型

实现欢迎、专家选择、三种模式、七位专家基本对话、临时会话、流式回复、清除、导出和基础安全路由。

### M2：封闭测试

完成自动化测试、角色盲测、红队测试、无障碍与移动端优化、成本和错误监控，以及隐私文本和危机资源人工审核。

### M3：公开 MVP

小流量发布，启用人格版本管理、匿名质量反馈和模型回归监控，并根据数据决定是否进入知识增强阶段。

### M4：可选知识增强

接入经过授权或公版的资料，提供可核验的阅读来源，并评估增加多流派观点比较功能。

## 11. MVP 验收标准

- 七位专家全部通过人格配置校验。
- 核心流程端到端测试通过。
- S2/S3 测试用例全部退出角色并采用相应安全响应。
- 人工盲测角色识别率不低于 75%。
- 评测集中严重理论串用为零。
- 常规日志中不存在完整对话内容。
- 页面刷新后无法恢复旧会话。
- 隐私、安全和历史真实性说明均可访问。

## 12. 尚待实施前确认的运营事项

以下事项不阻塞代码架构设计，但阻塞公开上线：确定模型供应商及数据处理条款；确定首发地区并核验当地危机资源；确认肖像、文本与译名使用权；指定具备心理学背景的内容评审者。
